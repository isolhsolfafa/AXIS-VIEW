# APS Lite 기획서

> AXIS-VIEW 차세대 생산계획 시스템
> 작성일: 2026-03-27
> 상태: 기획 단계

---

## 1. 시스템 개요

### 1.1 목적
CRM → SAP → AXIS 간 생산계획 데이터 흐름을 자동화하고,
맨먼스 기반 자동산출 + 협력사 성적지표 기반 자동분할로
생산계획 수립 효율을 높인다.

### 1.2 데이터 흐름 (전체 파이프라인)

```
[CRM 영업] ──생산계획──▶ [SAP ERP] ──초기계획──▶ [AXIS APS Lite]
                                                    │
                                          ┌─────────┴─────────┐
                                          ▼                   ▼
                                   맨먼스 자동산출      협력사 성적평가
                                          │                   │
                                          └─────────┬─────────┘
                                                    ▼
                                             자동분할 (배분)
                                                    │
                                                    ▼
                                          관리자 커스텀마이징
                                                    │
                                                    ▼
                                            확정 생산계획
                                                    │
                            ┌───────────────────────┤
                            ▼                       ▼
                     [공장 생산실행]          [SAP 실적동기화]
                     실시간 데이터 수집              │
                     실적처리                        ▼
                            │                [CRM 출하현황]
                            └──────────────────────▶│
                                                    ▼
                                             영업 출하확인
```

### 1.3 현재 상황
- OPS 설비 5대 가동 중, Q2까지 100% 전환 예정
- 월 170~200대 생산 규모
- SAP RFC 연동 필요 (사내서버 마이그레이션 예정)
- 현재 인프라: Railway(BE) + Netlify(FE) → 사내서버 전환 필요

## 2. 인프라 전환 계획

### 2.1 현재 vs 목표

| 항목 | 현재 | 목표 |
|------|------|------|
| BE (AXIS-OPS) | Railway (클라우드) | 사내서버 |
| FE (AXIS-VIEW) | Netlify | 사내서버 or Netlify 유지 |
| DB | Railway PostgreSQL | 사내서버 PostgreSQL |
| SAP 연동 | 불가 (RFC 접근 안됨) | 직접 RFC 호출 가능 |
| QMS 연동 | ETL 수동 | 사내 API 직접 연동 |

### 2.2 SAP RFC 연동 아키텍처

```
[AXIS-OPS BE (사내서버)]
    │
    ├── pyrfc / JCo ──▶ [SAP RFC] ──▶ 수주데이터 수신
    │                                   실적데이터 전송
    ├── REST API ──▶ [AXIS-VIEW FE]
    │
    └── DB 직접 ──▶ [QMS / 레거시 시스템]
```

### 2.3 마이그레이션 우선순위
1. 사내서버 환경 구축 (OS, Docker, PostgreSQL, Python)
2. AXIS-OPS BE 마이그레이션 + 검증
3. SAP RFC Gateway 모듈 개발 (pyrfc)
4. FE 배포 경로 결정 (사내 Nginx vs Netlify 유지)
5. QMS 인터페이스 연동

APS Lite

### 3.3 [M2] 맨먼스 자동산출

#### 3.3.1 표준공수 테이블

OPS에 쌓인 작업시작/완료 데이터에서 역산한다.

**데이터 소스 (현재 OPS 보유)**
- `app_task_details` — 작업 시작/완료 시간, task_category별
- 공정: MECH, ELEC, TMS(L), TMS(R), PI, QI, SI

**산출 로직**
```
표준공수[모델][공정] = PERCENTILE(실제소요시간, 0.7)

* 중앙값(50%) 대신 70% 백분위를 사용하는 이유:
  - 중앙값은 숙련공 기준이라 현실과 괴리
  - 평균은 이상치에 취약
  - 70%는 "대부분의 작업자가 달성 가능한 시간"
  - 현장 데이터 축적에 따라 60~80% 범위에서 조정 가능
```

**표준공수 테이블 예시**
```
모델      | MECH  | ELEC  | TMS(L)| TMS(R)| PI   | QI   | SI   | 합계
----------|-------|-------|-------|-------|------|------|------|------
GST-A     | 16h   | 12h   | 4h    | 4h    | 3h   | 2h   | 2h   | 43h
GST-B     | 20h   | 16h   | 5h    | 5h    | 4h   | 3h   | 2h   | 55h
GST-DUAL  | 18h   | 14h   | 4h    | 4h    | 3h   | 2h   | 2h   | 47h
```
> 위 수치는 예시이며, 실제 OPS 데이터 로드 후 산출 필요

#### 3.3.2 월간 필요공수 계산

```
월간 필요공수 = SUM(수주모델별 수량 x 표준공수[모델])

예시: 4월 수주
  GST-A  80대 x 43h = 3,440h
  GST-B  50대 x 55h = 2,750h
  ─────────────────────────
  합계                6,190h
  맨먼스 환산 = 6,190 / (8h x 22일) = 약 35.2 M/M
```

#### 3.3.3 월간 가용공수 계산

```
가용공수[협력사] = 투입인원 x 일일근무시간 x 월가동일수

예시:
  협력사A (FNI)  : 6명 x 8h x 22일 = 1,056h
  협력사B (BAT)  : 8명 x 8h x 22일 = 1,408h
  협력사C (C&A)  : 5명 x 8h x 22일 =   880h
  ───────────────────────────────────────
  총 가용                              3,344h
```

**검토 필요 사항**
- [ ] OPS 공수 데이터 로드 상태 확인 — 어떤 테이블에 어떤 형태로?
- [ ] 현재 모델 분류 체계 — 모델명 기준이 S/N prefix? product_info?
- [ ] DUAL 모델 TMS L/R 분리 기준
- [ ] 투입인원 데이터 소스 — 출퇴근(attendance) 기반? 수동 입력?

### 3.4 [M3] 자동분할 (협력사 배분)

#### 3.4.1 성적지표 가중 배분

**현재 OPS 보유 데이터 (PartnerEvaluation)**
```
평가항목:
  - defect_rate  (불량률)
  - nan_rate     (누락률)
  - score        (종합점수 0~10)
  - finalGrade   (등급 A~E)
```

**배분 공식**
```
가중점수[협력사] = (납기준수율 x W1) + (품질점수 x W2) + (가용능력비 x W3)

  W1 = 0.35 (납기)
  W2 = 0.40 (품질)
  W3 = 0.25 (생산능력)

  * 가중치는 관리자가 조정 가능
  * 품질점수 = OPS PartnerEvaluation score 활용
```

**배분 로직**
```
1. 각 협력사 가중점수 산출
2. 비율 계산: 배분비[i] = 가중점수[i] / SUM(가중점수)
3. 물량 배분: 배분수량[i] = ROUND(총수량 x 배분비[i])
4. 가용상한 체크: 배분수량[i] x 표준공수 <= 가용공수[i]
5. 초과 시 → 남은 협력사에 재분배 (재귀)
6. 단수 조정: 합계 = 총수량 되도록 보정
```

#### 3.4.2 분할 제약 조건
- 최소 배분 수량 (예: 5대 미만이면 배분 안 함)
- 특정 모델-협력사 고정 배정 (관리자 설정)
- 긴급 오더 우선 배정
- 기구/전장 협력사 분리 (MECH vs ELEC 별도 배분)

**검토 필요 사항**
- [ ] 납기준수율 데이터 — 현재 OPS에서 산출 가능한가?
- [ ] 기구/전장 협력사 구분 — production.ts의 mech_partner, elec_partner 활용
- [ ] 배분 단위 — O/N 단위? S/N 단위?

## 4. 데이터 활용 전략

### 4.1 즉시 활용 가능 (통계 기반)

| 활용 | 데이터 소스 | 방법 | 결과물 |
|------|------------|------|--------|
| 표준공수 산출 | app_task_details | 70% 백분위 집계 | 모델x공정 공수 테이블 |
| 협력사 성적 | PartnerEvaluation | 자동 집계 | 가중치 배분 테이블 |
| 납기 예측 | 수주+리드타임 이력 | 선형회귀 | 예상 완료일 |
| 이상 탐지 | 공수 데이터 | IQR / Z-score | 비정상 작업 알림 |

### 4.2 중기 활용 (간단한 ML)

| 활용 | 방법 | 기대효과 |
|------|------|---------|
| 납기 예측 고도화 | Random Forest | 물량+인원+모델 복합 예측 |
| 공수 이상 탐지 | Isolation Forest | 비정상 패턴 자동 감지 |
| 수요 예측 | ARIMA / Prophet | 월별 수주량 예측 |
| 학습곡선 반영 | 지수 감쇠 모델 | 숙련도 향상분 표준공수 반영 |

### 4.3 장기 활용 (LLM 인터페이스)

```
관리자: "4월 FNI 납기준수율 알려줘"
LLM → SQL 변환 → 결과 반환: "4월 FNI 납기준수율 92%, 전월 대비 3%p 상승"

관리자: "다음 달 GST-A 80대 배분 시뮬레이션 해줘"
LLM → APS 엔진 호출 → 분할 결과 + 가용 여부 반환
```

## 6. 열린 질문 (같이 고민할 것들)

### Q1. 표준공수 산출 기준
현재 OPS에 쌓인 데이터 양이 모델별로 충분한가?
최소 30건 이상의 완료 데이터가 있어야 백분위 산출이 의미 있다.
데이터가 부족한 모델은 유사 모델에서 보정하거나 현장 경험치를 초기값으로 설정해야 한다.

### Q2. 가동일수 / 투입인원 관리
현재 투입인원은 어디서 관리하는가?
출퇴근(attendance) 데이터에서 자동 산출 vs 별도 인력계획 입력?
가동일수는 공장 캘린더(휴일, 특근) 관리가 필요한가?

### Q3. 배분 단위
O/N(주문) 단위로 배분인가, S/N(제품) 단위로 배분인가?
O/N 단위가 관리 편의성은 높지만, 같은 주문 내에서 협력사를 나눌 필요가 있는지?

### Q4. 기구/전장 분리 배분
MECH(기구)와 ELEC(전장) 협력사가 다른 경우,
물량 배분을 공정별로 따로 해야 하는가?
현재 production.ts에 mech_partner, elec_partner가 분리되어 있음.

### Q5. SAP 연동 범위
초기에는 어디까지 연동할 것인가?
수주 수신만 먼저? 실적 전송도 동시에?
RFC 함수 목록을 SAP 담당자에게 사전 확인 필요.

### Q6. 엑셀 데이터 전환 시점
현재 리얼타임 관리 중인 엑셀 데이터를 DB로 완전 전환하는 시점은?
전환 전까지 엑셀과 DB 이중 관리가 발생할 수 있음.

**OPS BE (API)**
- 작업 시작/완료 실시간 데이터
- S/N별 공정 진행률 (MECH, ELEC, TMS, PI, QI, SI)
- 협력사 배정 정보 (mech_partner, elec_partner)
- 출퇴근 데이터 (attendance)
- QR 기반 제품 추적

**VIEW FE (대시보드)**
- 공장 대시보드 (KPI, 월간실적)
- 협력사 평가 (score, grade)
- 생산 실적 관리 (confirm/cancel)
- 사용자 분석

**ETL**
- 불량 데이터 (defect_record) — QMS/엑셀 기반 적재
- 분석 스키마 (analytics) — 접속 로그 집계
